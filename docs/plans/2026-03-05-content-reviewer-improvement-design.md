## 一、整体结构：双模式设计

### 模式触发逻辑

```
用户调用 content-reviewer
        ↓
检测触发词：
  "快速" / "quick" / "速览" / "看一眼"  → 快速模式（默认）
  "完整" / "full" / "详细" / "深度"    → 完整模式
  未指定                               → 快速模式（默认）
        ↓
快速模式完成后，综合评分 < 42/70
        → 自动提示"建议切换完整模式"
```

### 两个模式的边界

| 步骤 | 快速模式 | 完整模式 |
|------|---------|---------|
| 量化预检 | 执行 | 执行 |
| 通读全文 | 执行 | 执行 |
| 7 维评分 | 每维度只列 1 个最严重扣分点 | 所有扣分点 |
| 事实验证 | 跳过 | 仅高风险内容，附截图证明 |
| 修改清单 | 仅必须项，最多 5 条 | 必须/建议/可选全部 |
| AI 痕迹 | 统计命中数量，不逐词列出 | 逐词列出（精确到段落）|
| 联动出口 | 显示 | 显示 |

### 总分统一

- 总分：**70 分**（7 维 × 10 分）
- 发布阈值：**≥ 49/70**（70%）
- CLAUDE.md 同步更新（原"≥48/60"改为"≥49/70"）

## 二、评分标准精修（A 全面）

### A1. AI 痕迹检测——强化版

1. **保留全部四级红旗词库**，不删减
2. **新增检测维度**：
   - 句子开头模式：「当然」「显然」「不难看出」
   - 数字列举偏好：「有以下三点」「主要包括以下几个方面」
   - 语义重复：同一段内意思相近的句子出现 2 次以上
3. **一级红旗扣分提升**：-1 → **-1.5**
4. **结构性痕迹权重提升**：段落雷同/过度列举 -0.5/-1 → **-1.5/-2**
5. **上限不变**：最低 1 分（避免负分）
6. **完整模式**：逐词列出命中位置（精确到段落）

### A2. 原创性评分——增加正向锚定

增加正向描述（不改变总分上限，用于锚定判断）：

> 8-10 分的文章通常包含：作者亲历的具体案例、与主流观点不同的判断、让人「哦原来如此」的类比

### A3. 可读性/逻辑评分——文章类型感知豁免

| 文章类型 | 自动识别特征 | 豁免规则 |
|---------|------------|---------|
| 教程/手把手 | 含「步骤」/「Step」/有序列表 | 结构雷同不扣分 |
| 观点/评论 | 第一人称多、无代码块 | 代码/正文比不计算 |
| 技术对比 | 含对比表格 | 过度列举不扣分 |

### A4. 总分一致性

- SKILL.md 总分统一为 70 分
- 发布阈值 ≥ 49/70
- CLAUDE.md 中描述同步修正

## 三、事实准确性验证——截图证明（A 强化）

### CLI/代码命令执行层级

```
文章中出现 CLI 命令 / 代码示例
        ↓
1. 检查本地是否有该工具（which / command -v）
        ↓ 有 → 直接执行，捕获输出
        ↓ 无
2. Docker 运行：docker run --rm <镜像> <命令>
        ↓ 成功 → 捕获输出
        ↓ 失败
3. 标记 [需要人工验证]
```

### 执行结果截图

1. 将 stdout/stderr 写入临时 HTML（终端样式：黑底白字）
2. webapp-testing（Playwright）渲染并截图
3. 保存至 `/tmp/review-evidence/<命令名>-<时间戳>.png`
4. 报告中引用截图路径

**HTML 模板**：
```html
<div style="background:#1e1e1e;color:#d4d4d4;font-family:monospace;
            padding:16px;border-radius:8px;white-space:pre-wrap">
$ <命令>
<输出内容>
</div>
```

### 安全约束

- 只执行只读、无副作用命令（`--help`、`--version`、`list`、`get` 类）
- 不执行写操作、删除、网络请求类命令（标注「涉及副作用，跳过执行」）
- Docker 使用 `--rm` 自动清理

### 验证报告格式

```markdown
#### 验证项：`docker --version`
- 验证方式：本地执行
- 截图：`/tmp/review-evidence/docker-version-20260305.png`
- 结论：✅ 版本与文章描述一致

#### 验证项：`kubectl apply -f deployment.yaml`
- 本地检查：kubectl 未安装
- 降级方式：Docker bitnami/kubectl:latest
- 截图：`/tmp/review-evidence/kubectl-apply-20260305.png`
- 结论：❌ 命令报错，文章缺少 kubeconfig 前置说明 → 扣 -2
```

## 四、输出格式改进（B 全面）

### B1. 修改清单质量要求

每条修改建议必须满足"可直接替换"标准：

```markdown
1. **AI 一级红旗词：「综上所述」**
   - 位置：第 8 段开头
   - 修改前：`综上所述，Claude 4 在以下三个方面表现突出...`
   - 修改后：`回头看这三个维度——速度、上下文、工具调用——Claude 4 的提升是系统性的，不是某个单点的优化。`
   - 说明：去掉总结套话，改为有态度的收尾句
```

禁止出现"建议修改此处"、"可以考虑优化"等无实质内容的表述。

### B2. 快速模式输出格式

```markdown
# 快速审查报告

## 量化预检
[标准表格]

## 综合评分
| 维度 | 分数 | 最主要问题 |
|------|------|-----------|
| 可读性 | 7/10 | 第 4 段超过 200 字，需拆分 |
| ... | ... | ... |
| **综合** | **X/70** | 状态标识 |

## Top 5 必须修改
[仅列必须项，最多 5 条，每条含改前→改后]

## 下一步
> 综合分 X/70 [低于/接近/超过] 发布线（49/70）
> 建议：输入「完整审查」获取详细报告 + 全部修改清单
```

### B3. 完整模式末尾——直接修改出口

```markdown
---
## 是否需要帮你修改？

输入以下指令继续：
- `修改必须项` — 只改必须修改的内容
- `修改全部` — 改必须+建议所有项
- `修改第 N 条` — 只改指定条目
```
