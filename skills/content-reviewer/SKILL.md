# Content Reviewer

**6 维评分审查文章质量，输出修改清单和发布建议**

---

## 使用方式

1. 用户提供文章（Markdown 文件路径 或 直接贴内容）
2. 执行 6 维评分审查
3. 输出评分报告 + 修改清单 + 发布建议

---

## 审查流程

### 第一步：通读全文

- 用 Read 工具读取文章完整内容
- 记录第一遍阅读的直觉感受：流畅度、信息密度、是否想读下去
- 标记需要细查的段落

### 第二步：6 维逐项评分

每个维度 1-10 分，附具体扣分点。

---

## 6 维评分标准

### 维度 1：可读性（Readability）

**评判要素**：

| 检查项 | 扣分标准 |
|--------|---------|
| 段落长度 | 超过 150 字的段落每个 -0.5 |
| 冗余表达 | 每处"其实就是""说白了就是""简单来说"等重复解释 -0.3 |
| 长句拆分 | 超过 60 字的单句每个 -0.5 |
| 节奏感 | 连续 3 段以上结构雷同（如全是"首先...其次...最后"）-1 |
| 过渡衔接 | 段落之间缺少承接，跳跃感明显，每处 -0.5 |
| 代码/正文比 | 代码块占比超过 50% 且缺少解释 -1 |

**评分锚点**：
- 9-10：一口气读完，没有卡壳的地方
- 7-8：整体流畅，个别段落可以精简
- 5-6：有些地方需要重读才能理解
- 3-4：频繁卡壳，信息密度不均
- 1-2：难以读完

### 维度 2：逻辑连贯性（Logic Flow）

**评判要素**：

| 检查项 | 扣分标准 |
|--------|---------|
| 论点支撑 | 核心论点缺少论据支撑 -2 |
| 论证链条 | 推理跳跃（A→C，缺少 B）每处 -1 |
| 信息完整 | 提到概念但未解释，读者可能不懂的，每处 -0.5 |
| 前后矛盾 | 前后表述不一致，每处 -2 |
| 结构呼应 | 开头承诺的内容在正文中未兑现 -2 |
| 章节顺序 | 章节排列不符合认知递进（如先讲高级用法再讲基础）-1 |

**评分锚点**：
- 9-10：论证严密，读完对主题有完整认知
- 7-8：整体连贯，个别地方可以补充
- 5-6：有断裂感，部分论点支撑不足
- 3-4：逻辑混乱，读完不知道重点
- 1-2：东拉西扯，无主线

### 维度 3：标题与 Hook（Title & Hook）

**评判要素**：

| 检查项 | 扣分标准 |
|--------|---------|
| 标题字数 | 不在 15-25 字范围 -1 |
| 关键词覆盖 | 标题缺少核心关键词 -1 |
| 利益承诺 | 标题没有明确的读者收益 -1 |
| 标题兑现 | 正文无法兑现标题承诺 -3 |
| Hook 吸引力 | 开头 100 字无法抓住注意力 -2 |
| Hook 关联度 | 开头与正文核心内容脱节 -1 |

**评分锚点**：
- 9-10：看到标题就想点，开头 3 秒就被抓住
- 7-8：标题有吸引力，开头能留住人
- 5-6：标题平平，开头没有特别抓人
- 3-4：标题模糊，开头冗长
- 1-2：标题与内容不匹配

### 维度 4：事实准确性（Factual Accuracy）

**评判要素**：

| 检查项 | 扣分标准 |
|--------|---------|
| 代码正确性 | 代码示例有语法错误或逻辑错误，每处 -2 |
| 命令准确性 | CLI 命令参数错误或已废弃，每处 -2 |
| 版本信息 | 提到的版本号/发布日期不准确，每处 -1 |
| 数据引用 | 引用的数据无来源或来源不可靠，每处 -1 |
| 链接有效性 | 外部链接失效（如有能力验证），每处 -0.5 |
| 概念准确 | 技术概念解释有误，每处 -2 |

**重要**：如果无法验证某些事实，标记为 `[需要验证]` 而不是瞎猜。用 WebSearch 验证可疑的技术声明。

**评分锚点**：
- 9-10：所有技术细节经得起推敲
- 7-8：整体准确，个别细节可以更精确
- 5-6：有几处需要修正的错误
- 3-4：多处事实错误，影响可信度
- 1-2：大量错误，不可发布

### 维度 5：AI 痕迹（AI Trace Detection）

**红旗词库**（出现即扣分）：

```
一级红旗（每个 -1）：
在当今...的时代 | 随着...的发展 | 让我们一起来看看 | 总而言之
综上所述 | 毋庸置疑 | 值得注意的是 | 不难发现

二级红旗（每个 -0.5）：
赋能 | 助力 | 一站式 | 全方位 | 深度赋能 | 无缝衔接
生态 | 闭环 | 抓手 | 底层逻辑 | 顶层设计 | 方法论
打通 | 沉淀 | 拉齐 | 对齐 | 颗粒度

三级红旗（每个 -0.3）：
首先...其次...最后（连续使用）| 不仅...更是... | 一方面...另一方面...
事实上 | 实际上 | 需要指出的是 | 重要的是
```

**结构性 AI 痕迹**：

| 检查项 | 扣分标准 |
|--------|---------|
| 段落结构雷同 | 连续 3+ 段"概念→解释→示例"结构 -1 |
| 过度列举 | 每个论点都用 3 点或 5 点列举 -1 |
| 语气均匀 | 全文无情绪波动，像百科全书 -1 |
| 缺少个人视角 | 没有"我"的经验、观点或态度 -1 |
| 结尾套路 | "希望本文对你有帮助" 类结尾 -1 |
| 过渡词过多 | 几乎每段都以"此外""另外""同时"开头 -0.5 |

**评分锚点**：
- 9-10：像是真人写的经验分享，有温度
- 7-8：整体自然，少量痕迹可修
- 5-6：能感觉到 AI 味，但还能接受
- 3-4：明显 AI 生成，读者会有不适感
- 1-2：一眼 AI，不可发布

### 维度 6：平台适配（Platform Fit）

**默认目标平台**：微信公众号。如果用户指定了其他平台，按对应标准审查。

**微信公众号标准**：

| 检查项 | 扣分标准 |
|--------|---------|
| 全文字数 | < 1500 字内容太薄 -1，> 5000 字过长 -1 |
| 外部链接 | 微信不支持外链，每处外链 -0.5（应改为文末引用） |
| 图片密度 | 纯文字超过 800 字没有图/表/代码块 -1 |
| 代码块长度 | 单个代码块超过 30 行 -1（手机阅读体验差） |
| 标题层级 | 超过 3 级标题 -0.5（手机端层级过深难阅读） |
| 首图 | 缺少封面图建议 -0.5 |
| 摘要 | 缺少 120 字以内的摘要 -0.5 |

**评分锚点**：
- 9-10：完全适配微信阅读体验
- 7-8：整体适配，个别地方需要调整
- 5-6：有几处不适配，需要修改
- 3-4：排版不适合手机阅读
- 1-2：完全不适合目标平台

```markdown
## 修改清单

按优先级排列，标注影响分值：

### 🔴 必须修改（影响发布质量）
1. [问题] — 位置：[段落/行号] — 影响：[扣分值] — 建议：[具体修改方案]
2. ...

### 🟡 建议修改（提升阅读体验）
1. [问题] — 位置：[段落/行号] — 建议：[具体修改方案]
2. ...

### 🟢 可选优化（锦上添花）
1. [建议] — 理由：[为什么]
2. ...

---

## 发布建议

**建议**：🟢 可以发布 / 🟡 修改后发布 / 🔴 需要大改

[1-2 句总结性建议]
```

---

## 评分阈值与建议

| 综合评分 | 建议 |
|---------|------|
| 48-60 | 🟢 可以发布，做可选优化即可 |
| 36-47 | 🟡 修改必须项后可发布 |
| 24-35 | 🔴 需要较大修改，建议逐维度改进 |
| < 24 | ⛔ 建议重写 |

---

## 注意事项

- 所有评分必须有具体依据，不打印象分
- 事实准确性维度：不确定的内容标 `[需要验证]`，可用 WebSearch 验证可疑声明
- AI 痕迹维度：只标记实际命中的红旗词和结构痕迹，不要过度解读
- 修改清单中的建议必须具体可操作——给出修改后的文本，而不只是说"建议修改"
- 如果文章整体质量很高，不要为了找问题而找问题
- 审查完成后，如果用户同意修改，可以直接帮助修改文章
