# 写作风格指南 - 技术博客版

## 评分优化规则（对齐 content-reviewer 6 维评分）

### 标题公式（标题与 Hook 维度，目标 9+）

**硬性规则**：
- 字数：15-25 字（不含标点）
- 必须包含核心技术关键词
- 必须有明确读者收益（时间承诺、效果承诺、痛点解决）

**公式**：`[时间/数量] + [动作] + [技术关键词] + [收益]`

| 类型 | 差的标题 | 好的标题 |
|------|---------|---------|
| 快速入门 | "Docker 入门" (5字) | "5 分钟用 Docker 部署你的第一个 Web 应用" (19字) |
| 实战教程 | "uv 使用教程" (6字) | "uv 实战教程：用 Rust 速度管理 Python 项目" (19字) |
| 深度解析 | "Go 内存管理" (6字) | "Go GC 调优实战：从 200ms 停顿降到 5ms 的全过程" (22字) |

### Hook 开头（标题与 Hook 维度，目标 9+）

**开头 100 字必须包含三要素**：痛点/场景 → 解决方案 → 阅读价值

**模板 A：痛点切入**
```
[具体痛点描述，1-2 句] + [工具/方案名] + [一句话说明它怎么解决] + [本文覆盖范围]
```
示例："pip 装包慢、venv 命令长、pyenv 配置烦——这三个痛点困扰 Python 开发者多年。uv 用一个二进制文件解决了全部问题。"

**模板 B：场景切入**
```
[读者会遇到的真实场景] + [现有方案的不足] + [本文方案的优势]
```
示例："团队里每个人的 Python 环境都不一样，requirements.txt 写了 200 行还是装不上。锁文件本该解决这个问题，但 pip 不支持。"

**模板 C：数据切入**
```
[一个反直觉的数据/事实] + [为什么会这样] + [本文将展示什么]
```
示例："同一个项目，pip install 用了 45 秒，uv 只用了 2.1 秒——快了 20 倍。这不是营销数据，是我本机实测的结果。"

**禁止**：
- ❌ "在当今...的时代" / "随着...的发展" 开头
- ❌ 以定义开头："XXX 是一个..."
- ❌ 反问句钓鱼："你是否也有这样的困扰？"

### 反 AI 结构痕迹（AI 痕迹维度，目标 9+）

**段落结构变化规则**：
- 连续段落禁止使用相同结构。如果上一段是"概念→解释→代码"，下一段必须换结构
- 可选结构：代码先行→逆向解释、对话体提问→回答、经验教训→原理、对比表→结论

**个人视角插入点**（每篇至少 2 处）：
- 踩坑经验："我在迁移旧项目时发现——"
- 选择理由："选 uv 而不是 poetry 的原因很简单——"
- 评价判断："这个功能设计得很克制，只做了该做的事"
- 实测数据："本机实测，冷启动 2.1 秒"

**段首词多样化**：
- 禁止连续 2 段以"此外""另外""同时""值得注意的是"开头
- 用具体内容开头替代过渡词：直接说下一件事，不要"另外，还有一个功能..."

**结尾规则**：
- ❌ "希望本文对你有帮助"
- ❌ "如果有问题欢迎留言"
- ✅ 以具体下一步行动结束："装好 uv 后，在现有项目里跑一次 `uv pip install -r requirements.txt`，体感一下速度差异。"
- ✅ 以技术展望结束（不超过 2 句）："uv 的 workspace 功能还在快速迭代，monorepo 支持值得关注。"

### 平台适配规则（平台适配维度，目标 9+）

**微信公众号适配**：
- `description` 字段必填，≤120 字，作为微信摘要
- 外部链接集中在参考资料区，正文中用「参见文末链接」替代内联 URL
- 单个代码块 ≤30 行（手机阅读），超长代码拆分为多个块并加解释
- 纯文字段落不超过 800 字必须插入图/表/代码块打破单调
- 标题层级 ≤3 级（##、###、####），不要用 #####

**可读性节奏**：
- 段落 ≤150 字，超过则拆分
- 代码块之间至少 2-3 句解释文字，禁止连续两个代码块无间隔
- 长句 ≤60 字，超过则断句
- 每 400-600 字插入一张节奏图（截图或 AI 图）

**示例**：
```python
def quick_sort(arr: list[int]) -> list[int]:
    """
    快速排序算法实现

    Args:
        arr: 待排序的整数列表

    Returns:
        排序后的列表
    """
    if len(arr) <= 1:
        return arr

    pivot = arr[len(arr) // 2]
    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]

    return quick_sort(left) + middle + quick_sort(right)
```

## 🔗 链接规范

### ✅ 正确格式

**明链接**（推荐）：
```markdown
- **Ollama 官网**: https://ollama.com
- **GLM4 GitHub**: https://github.com/THUDM/GLM-4
```

### ❌ 错误格式

**Obsidian 内部链接**（不要用）：
```markdown
- [[Ollama 官方文档]]  ❌
- [[GLM4 使用指南]]    ❌
```

### 📌 参考链接章节

**只保留一个资源章节**，链接数量 ≤5 条时用列表，>5 条时用表格（微信公众号中表格作为单个 HTML 元素渲染，不会被广告分割）：

**≤5 条链接（列表格式）**：
```markdown
## 参考资料

- **资源名称1**: https://example.com
- **资源名称2**: https://example.com/docs
```

**>5 条链接（表格格式，推荐）**：
```markdown
## 参考资料

| 项目 | 链接 |
|------|------|
| 资源名称1 | https://example.com |
| 资源名称2 | https://example.com/docs |
| 资源名称3 | https://github.com/example/repo |
```

**删除这些冗余章节**：
- ❌ "相关资源" + "参考链接"（二选一）
- ❌ 使用 callout 包裹链接（直接列出）
